# Futures Convexity Framework Architecture Audit

Date: 2026-05-14

## Executive Summary

The repository already contains an internal futures data and screening stack under `src/volbook/` and `scripts/volbook/`. That stack should be treated as the starting point for any institutional futures convexity/regime framework. It includes IBKR historical futures connectors, a DuckDB 1-minute lake, partitioned Parquet export, contract/universe metadata, continuous and dated futures ingestion workflows, and a discretionary futures dashboard.

The current stack is suitable as a reusable data foundation and compatibility boundary, not as the finished research framework. It does not yet provide a validated CL 4-hour research dataset, back-adjusted continuous futures, product-specific roll calendars, term-structure feature engineering, leakage-safe walk-forward modeling, portfolio construction, execution simulation, or institutional convexity/tail evaluation. The next phase should extend this stack rather than duplicate it.

## TECH STACK

Existing code already uses Python 3.12 with `polars`, `duckdb`, `pandas`, `numpy`, `scipy`, `statsmodels`, `scikit-learn`, `matplotlib`, `plotly`, optional `ib_insync`, and optional `TA-Lib`. The proposed convexity framework stack can build on this and add `pyarrow`, `xgboost`, `lightgbm`, `hmmlearn`, and later `pytorch/lightning/transformers` only when research needs justify them.

## EXISTING INTERNAL DATA INFRASTRUCTURE and DATA INTEGRATION REQUIREMENTS

Before implementing any new futures research, modeling, or execution framework, an agent must first locate and audit the existing internal futures database and futures screen infrastructure in this repository. Start with `src/volbook/`, `scripts/volbook/`, `tests/test_volbook_*`, and the generated-data paths documented below. Do not create a parallel futures lake, contract registry, IBKR connector, roll engine, or dashboard bundle until the existing interfaces have been evaluated for reuse.

The new framework must preserve compatibility with the current futures screen. It should read from or extend the existing `MinuteLake`, Parquet export, contract metadata, and bundle/dashboard outputs where appropriate. Any new CL 4-hour, 1-hour, or daily research dataset should be derived from the internal lake or exported Parquet, with explicit lineage back to `symbol`, `expiry`, timestamp, roll rule, source table, and refresh job. If a required capability is missing, add it as a well-scoped extension around the existing data layer rather than replacing the data infrastructure silently.

The framework is not a retail technical indicator project, RSI-threshold bot, moving-average crossover system, deterministic indicator bot, or naive trend follower. It is intended to behave like a systematic macro volatility desk, CTA convexity engine, and latent-state forecasting framework. The objective is not predicting average returns; it is identifying latent instability, volatility expansion, directional persistence conditional on expansion, convex asymmetric opportunities, and metastable states before large moves.

## Current Architecture

### Primary Futures Tool

The existing futures screen is `volbook`, described in `src/volbook/README.md` as a futures OHLCV tool for the volatility book. It has two related but distinct data products:

- A JSON dashboard bundle at `data/volbook/bundle.json`, generated by `src/volbook/bundle.py` and dashboard refresh scripts. It stores selected daily/hourly/other IBKR bar series, TA-Lib indicator snapshots, and heuristic setup rows for rendering.
- A DuckDB minute-bar lake at `../data/futures_market.duckdb`, managed by `src/volbook/datalake.py` and minute refresh/backfill scripts. This is the more appropriate foundation for systematic research because it has normalized tables, ingest state, dated contracts, and Parquet export.

### Database Locations

Documented locations:

- `../data/futures_market.duckdb`: DuckDB lake for 1-minute futures bars.
- `../data/futures_market_parquet/`: partitioned Parquet export of the DuckDB lake.
- `data/volbook/bundle.json`: JSON bundle used by the screen/dashboard.
- `artifacts/volbook/volbook-futures-ohlcv.html`: standalone HTML dashboard.
- Cursor canvas output: `~/.cursor/projects/<workspace>/canvases/volbook-futures-ohlcv.canvas.tsx`.

The repo `.gitignore` excludes `/data/`, `data/volbook/`, `artifacts/`, and `*.parquet`, so these data products are intentionally not source-controlled.

### DuckDB Schema

`src/volbook/datalake.py` owns the lake schema:

- `bars_1m(symbol, expiry, ts, o, h, l, c, v, fetched_at)`
- Primary key: `(symbol, expiry, ts)`
- `expiry='continuous'` stores IBKR `ContFuture` trailing-window bars.
- `expiry='YYYYMM'` stores dated futures contract bars.
- `ingest_state(symbol, expiry, earliest_ts, latest_ts, head_ts, last_run_at, notes)`
- Primary key: `(symbol, expiry)`

The lake wrapper exposes:

- `upsert_bars`
- `get_state`
- `list_states`
- `set_head_timestamp`
- `record_notes`
- `row_count`
- `stitch_continuous_series`

### Parquet Export

`scripts/volbook/export_lake.py` exports the lake to:

```text
../data/futures_market_parquet/
  bars_1m/symbol=ES/expiry=202606/data_0.parquet
  bars_1m/symbol=CL/expiry=202606/data_0.parquet
  ingest_state.parquet
```

The exporter snapshots the DuckDB file plus WAL to avoid writer lock contention, then writes hive-partitioned Parquet. It defaults to dated bars only and excludes `expiry='continuous'` unless `--include-continuous` is passed. This is the best external-consumer interface for the new research stack because Polars, DuckDB, pandas, and Arrow can read it concurrently without touching the live DuckDB file.

### IBKR Connectors

`src/volbook/ibkr_client.py` wraps `ib_insync` and supports:

- Dated `Future` OHLCV via `fetch_futures_ohlcv`
- IBKR `ContFuture` OHLCV via `fetch_continuous_futures_ohlcv`
- Continuous 1-minute trailing windows via `fetch_continuous_minute_bars`
- Dated 1-minute bars via `fetch_dated_minute_bars`
- Batched dated qualification via `qualify_dated_futures`
- Curve discovery via `discover_futures_curve`
- `reqHeadTimeStamp` for continuous and dated contracts
- Retry logic that distinguishes true empty responses from `reqHistoricalData` timeouts

This connector should be reused. New vendors such as Databento, Polygon, CME direct, or additional IBKR endpoints should be added behind a compatible loader interface rather than bypassing this registry and lake design.

### Timeframe Availability

Current persisted/researchable timeframes:

- `bars_1m` in DuckDB and Parquet for continuous and dated futures.
- Dashboard bundle defaults: `1 day` for `1 Y` and `1 hour` for `30 D`.
- CLI can request arbitrary IB bar sizes such as `5 mins`, but only the minute lake has a normalized database schema.

Missing for the requested CL framework:

- No explicit 4-hour table or loader exists.
- No session-aware 4-hour bar construction exists.
- No canonical daily/1-hour/4-hour aligned research panel exists.

Recommendation: derive `1h`, `4h`, and daily bars from the Parquet/DuckDB minute lake with an explicit session calendar and resampling contract, then persist derived bars as new versioned research tables or Parquet datasets with source lineage.

### Continuous Futures and Roll Methodology

There are two continuous concepts:

- IBKR `ContFuture` bars stored with `expiry='continuous'`. These are convenient for recent front-month views but are opaque and limited by IBKR's continuous-history constraints.
- `MinuteLake.stitch_continuous_series(symbol, roll_days_before_expiry=N)`, which builds an on-demand continuous front-month series from dated bars.

The stitched logic selects, at each timestamp, the smallest dated `expiry` whose roll point has not yet been reached. The roll point is computed as the first day of the expiry month minus `roll_days_before_expiry`. It excludes `expiry='continuous'` rows to avoid double-counting.

This is useful but not institutional-grade for CL research yet. It is an unadjusted front-month stitch with a coarse calendar roll rule. It does not currently handle:

- Product-specific last trade dates or first notice dates.
- Volume/open-interest roll triggers.
- Back-adjusted or ratio-adjusted continuous prices.
- Carry-preserving Panama adjustments.
- Explicit roll return decomposition.
- Calendar spread or term-structure continuity.
- Negative-price or contract specification edge cases for CL.

### Update Mechanisms and Lineage

Refresh/backfill scripts:

- `scripts/volbook/backfill_futures_minute.py`: backfills trailing continuous `ContFuture` 1-minute windows, default `30 D`.
- `scripts/volbook/refresh_futures_minute.py`: daily refresh for continuous 1-minute bars using trailing windows.
- `scripts/volbook/walk_dated_futures_minute.py`: deep-history backward walk over dated contracts, default `5` years, 30-day chunks.
- `scripts/volbook/refresh_dated_futures_minute.py`: daily forward refresh for active dated contracts and optional seeding of new expiries.
- `scripts/volbook/export_lake.py`: DuckDB-to-Parquet exporter.
- `scripts/volbook/refresh_core_futures.py`: dashboard bundle refresh for daily/hourly bars, dated curves, or continuous-only front views.
- `scripts/volbook/refresh_core_futures_daily.py`: daily-only wrapper.
- `scripts/volbook/refresh_front_futures.py`: continuous-only wrapper.

Lineage currently exists at the ingest-state level (`earliest_ts`, `latest_ts`, `head_ts`, `last_run_at`, `notes`) and in each bundle series (`symbol`, `expiry`, `exchange`, `bar_size`, `duration`, `what_to_show`, `fetched_at`). There is no dataset version, feature version, roll-rule version, vendor version, or research split metadata yet.

### Feature and Screening Infrastructure

`src/volbook/indicators.py` computes TA-Lib indicators across a single OHLCV series, primarily for dashboard inspection. `src/volbook/signals.py` builds heuristic trend, mean-reversion, and range-breakout setups gated by ADX. `src/volbook/html_writer.py` renders a self-contained dashboard with:

- Contract/timeframe selection.
- Technical chart with price, volume, ADX, realized vol, MACD, and RSI.
- Indicator tables.
- A discretionary watchlist.
- Liquidity/data-quality filters.
- Curve collapse by `(underlying, direction, strategy, timeframe)`.
- Direction conflict demotion.
- Excel export for visible opportunities.

The screening layer is explicitly documented as discretionary, not a systematic strategy. It should not be reused as a predictive model or labeled edge estimate. It can be reused for visualization, data-quality diagnostics, and compatibility with trader review.

## Reuse, Extend, Replace

### Reuse

Reuse these components directly:

- `FuturesSpec`, `KNOWN_FUTURES`, `CORE_MACRO_ALIASES`, `OPTIONS_UNDERLYING_ALIASES`, `month_pattern_for_root`, and `enumerate_dated_specs`.
- `IBHistoricalClient` for IBKR connectivity and qualification.
- `MinuteLake` schema and upsert/state operations as the canonical raw minute lake interface.
- `export_lake.py` Parquet layout for concurrent research consumption.
- `OhlcvBundle` only for dashboard compatibility, not as a primary research database.
- Existing tests around contracts, lake behavior, export, and dashboard string contracts.

### Extend

Add capabilities around the existing stack:

- A research data loader that reads `../data/futures_market_parquet/bars_1m/**/*.parquet` via Polars/DuckDB and can optionally query `MinuteLake`.
- A session-aware bar builder for CL `1h`, `4h`, and daily bars with explicit timestamp semantics.
- A product roll calendar module with CL-specific last trade, first notice, liquidity-roll, and roll-window rules.
- A back-adjusted continuous futures builder with transparent adjustment factors and a separate returns/carry decomposition.
- Derived datasets with explicit versioned paths, e.g. `../data/futures_research/features/...` or `../data/futures_research/bars/...`.
- Data validation: duplicate bars, missing intervals, stale bars, zero volume, contract gaps, roll discontinuities, split by RTH/GLOBEX if needed.
- Feature pipelines for volatility compression/expansion, trend persistence, jump/tail behavior, term structure, calendar spreads, cross-asset macro context, and regime state inference.
- Model/evaluation modules with purged walk-forward splits, embargo, feature importance, SHAP, and leakage tests.
- Reporting hooks that can render convexity/regime outputs into the existing dashboard or a new report without breaking Volbook.

### Replace or De-emphasize

Do not use these as research truth:

- TA-Lib dashboard indicators as alpha features without revalidation.
- ADX regime gate as the latent-state engine.
- `setup.confidence`, `Setup Grade`, or the discretionary watchlist score as calibrated probabilities.
- IBKR opaque `ContFuture` series as the sole source of historical research truth.
- The unadjusted `stitch_continuous_series` output as a back-adjusted research series without further adjustment and roll validation.

## Technical Debt and Bottlenecks

Key risks:

- IBKR pacing limits are severe for 1-minute multi-market history. Scripts sleep around 11 seconds per request to stay under small-bar pacing limits.
- DuckDB single-writer locking requires Parquet export for concurrent consumers.
- Continuous `ContFuture` requests cannot be chunked backward with `endDateTime`, limiting direct continuous history.
- Dated history is collected by many 30-day chunks across many contracts, which is slow and operationally fragile.
- Roll rules are coarse and not product-specific enough for CL institutional research.
- No canonical 4-hour resampling layer exists.
- No back-adjusted continuous contract table exists.
- No open-interest or volume-based roll selection exists.
- No cross-vendor data reconciliation exists.
- No feature store, model registry, dataset versioning, or experiment metadata exists.
- No leakage-safe research validation framework exists in the futures screen stack.
- Dashboard bundle data is JSON and convenient for UI, but not scalable or authoritative for research.

## Missing Institutional Capabilities

The new convexity/regime framework needs the following additions:

- CL continuous futures with explicit roll methodology and back-adjustment.
- Session-aware 4-hour primary bars, 1-hour secondary bars, and daily context bars.
- Research targets: normalized expansion magnitude `|P_{t+H}-P_t| / ATR_t`, directional persistence conditional on expansion, volatility expansion, tail-event probabilities, regime transition probabilities, and breakout continuation.
- Latent regime models: HMM, GMM, Bayesian state-space, change-point detection, and volatility clustering.
- Feature library: realized volatility, ATR compression, range compression, volatility-of-volatility, trend slope/acceleration, gap/jump behavior, liquidity/stale diagnostics, carry/curve, calendar spreads, CL vs DXY/rates/energy complex/equities/VIX/VX/inflation context.
- Modeling: logistic baselines, random forests, XGBoost, LightGBM, Bayesian models, and later deep learning only after tabular baselines are strong.
- Validation: expanding windows, walk-forward splits, purged CV, embargo, temporal feature availability checks, label leakage tests, and out-of-sample stability reports.
- Signal outputs: `expansion_prob`, `upside_prob`, `downside_prob`, `expected_move_atr`, `convexity_score`, `regime`.
- Portfolio/backtest analytics that emphasize positive skew, convexity, crisis alpha, tail participation, drawdown containment, hit rate conditional on expansion, and payoff asymmetry, not just Sharpe.
- Execution simulation that understands futures multipliers, tick sizes, session liquidity, slippage, roll costs, and contract selection.

## Recommended Target Integration

Start with a thin compatibility layer:

```text
src/futures_convexity/
  data/
    lake_loader.py          # reads volbook DuckDB/Parquet
    bar_builder.py          # 1m -> 1h/4h/daily session-aware bars
    continuous.py           # roll and back-adjustment logic
    validation.py           # missing/stale/duplicate/roll QA
  features/
  regimes/
  models/
  signals/
  portfolio/
  backtest/
  reports/
  configs/
tests/
```

Initial CL workflow:

1. Read CL dated bars from `../data/futures_market_parquet/bars_1m/symbol=CL/expiry=*/`.
2. Validate coverage and ingest freshness via `ingest_state.parquet`.
3. Build CL front-month and back-adjusted continuous bars with explicit roll configuration.
4. Resample to 4-hour primary bars plus 1-hour and daily secondary features.
5. Persist derived datasets with source lake path, export timestamp, roll rule, resampling rule, and code version.
6. Only then build latent-regime features, targets, and models.

This preserves the existing futures screen while allowing institutional research modules to mature separately.

## First Implementation Gate

Before any modeling framework code is written, complete these checks:

- Confirm that `../data/futures_market.duckdb` or `../data/futures_market_parquet/` exists in the user's runtime environment.
- Inspect `ingest_state` for `CL` dated and continuous coverage.
- Verify whether CL minute bars cover enough history for 4-hour labels at `H=4h`, `1d`, `3d`, and `5d`.
- Identify contract gaps and roll periods around CL monthly expiries.
- Decide whether the first research dataset uses IBKR only or requires Databento/CME direct for institutional coverage.
- Define the CL roll rule and back-adjustment method before any labels are computed.

## Bottom Line

The correct path is to reuse `volbook` as the internal futures data foundation, extend it with institutional continuous-contract and research-dataset layers, and avoid building a duplicate futures database. The current stack is a strong starting point for ingestion, registry, and screening, but the convexity/regime framework must add explicit roll methodology, session-aware 4-hour bars, back-adjustment, feature/target/version lineage, and leakage-safe modeling before it can support production-grade CL volatility expansion research.
